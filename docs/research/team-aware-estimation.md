# Team-Aware Effort Estimation (Human + AI)

**Status**: Implemented | **Category**: Novel — first executable model
**Location**: `core/agents/project-coordinator.md`

---

## Problem

Traditional effort estimation assumes a homogeneous team of human developers. Story points, developer-days, and t-shirt sizing all measure human effort. But in AI-augmented teams, the dynamics are fundamentally different:

- AI agents write code in minutes, not hours
- Multiple AI agents work in parallel at zero marginal cost
- Human review is the actual bottleneck — and it scales linearly, not in parallel
- The critical path is always human attention, not implementation capacity

No project management tool or AI coding framework provides an estimation model that accounts for these dynamics. Jira, Linear, Azure DevOps, and Shortcut all use traditional estimation. The Scrum.org "AI-Augmented Scrum Framework" (2026) identifies the problem but provides no executable solution.

## Solution

Every task in a project plan is tagged with its **executor type** and estimated accordingly:

| Executor | Symbol | Characteristics |
|----------|--------|----------------|
| Human only | `(human)` | Sequential, limited by attention, context switching cost |
| AI only | `(AI)` | Parallel, fast, needs clear specs, no domain intuition |
| Collaborative | `(human+AI)` | AI drafts, human reviews — bottleneck is the review |

### Estimation Template

```
Task                                    Executor    Effort      Bottleneck
─────────────────────────────────────────────────────────────────────────
Research existing solutions             AI          30 min      —
Architecture decision                   human+AI    1 hour      human review
Implementation (3 modules parallel)     AI          45 min      —
Unit tests                              AI          30 min      —
Human review of all AI output           human       2 hours     CRITICAL PATH
Integration testing                     human+AI    1 hour      human validation
Documentation                           AI          20 min      —
─────────────────────────────────────────────────────────────────────────
Total wall-clock (sequential):                      ~6 hours
Total wall-clock (with parallelism):                ~4 hours
Critical path:                                      human review + validation
```

### Key Principles

1. **Tag every task with the executor**: `(human)`, `(AI)`, or `(human+AI)`
2. **AI tasks are faster but need review**: AI writes code in minutes, human review adds time
3. **Parallel AI work is free**: Multiple agents simultaneously — don't estimate serially
4. **Human bottleneck is the real constraint**: 3 tasks needing review = 3x review time
5. **Research tasks need buffer**: External research has unpredictable duration — add 50%
6. **Never say "5 developer-days"** — say "4 hours wall-clock: 2h human review (critical path) + 2h parallel AI work"

## Impact

Project plans generated by the project-coordinator agent now show:
- Real wall-clock time (not fictional developer-days)
- Human bottleneck identified on critical path
- Parallel AI capacity utilized
- Clear ownership per task (who does what)

Example: A notification system integration estimated at "3-4 developer-days" traditionally becomes "4 hours wall-clock" when accounting for AI parallel work and identifying the 2.5-hour human review bottleneck.

### How It Works

1. User requests a plan (e.g., "how long to add VS Code adapter?")
2. **project-coordinator** applies the estimation template:
   - Tags each task: `(human)`, `(AI)`, `(human+AI)`
   - Identifies parallel AI tasks (free capacity)
   - Calculates wall-clock with and without parallelism
   - Marks human review as CRITICAL PATH
3. Output format: table with Executor, Effort, and Bottleneck columns
4. Never outputs "developer-days" — always wall-clock with bottleneck identified
